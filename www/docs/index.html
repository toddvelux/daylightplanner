1.2 Getting Started - Alternate Versions


The template bundles 4 version of the html files, each one of them has a separate subfolder in the "alternate" folder:

compressedUses a single optimized/compressed file for javascript/css and has image lazy-loading enabled. Best performances in terms of site loading speed.
nolazySame as the above but with lazy-loading turned off, useful if you want all your images to be indexed by search engines.
uncompressedSame as compressed version but it load individual, uncompressed css and js files. Useful if you want to debug the template or customize its javascript or CSS sources.
uncompressed-nolazySame as the uncompressed above but with lazy-loading turned off.
If using one of the alternative versions of this template, contained in the "alternate" folder, then all of the files and folder needed for that version are in the main root folder. Only the html files are contained in the "alternate" folder. These will need to be copied into the root folder to replace the current ones.


1.3 Getting Started - Lazy Loading


Lazy loading is a technique which defers loading of images: instead of loading all of them as soon as the page is requested, only the ones that are currently visible in the view port will be loaded. As soon as user scrolls the page and a previously hidden image will be visible, it will be loaded.

To take advantage of the feature, each image must be added like this:
<img src="img/blank/186x115.gif" data-original="img/content/port7-186x115.jpg" width="186" height="115" alt="alt text" />
Parameter List
srcSet the src parameter to a fake image (transparent gif) which has the same size of the original image. There are several ready made in the "img/blank" subfolder for different resolution. If you cannot find one that match your size, you will have to create it (to resize whatever of the sample blank images with will be ok). It's very important that the fake image has the exact size of the original image, or else something will most likely break.
data-originalreal url of the image

1.4 Getting Started - Website Layout Format


There are 2 supported layouts in the BigWig template. These are boxed layout and wide layout. By default the template will display the wide layout, but to achieve the boxed version, simply add the class boxed to the body tag of each HTML page in your website. Like this:
<body class="boxed">
Once this is done, the website content will now become a fixed width, centrally aligned 980px format. Once this happens the background of the site will be seen. If you wish to change the color of this background you can use the following snippet of CSS. Place it in the head section of each of the HTML pages. Just replace the "#fafafa" with your desired hex color reference.
<style>
    body.boxed{
        background-color:#fafafa;
    }
</style>

2.1 Portfolio - Columns


The columns format consists of a regular grid of image arranged in columns with spacing between the columns. Projects shown in this way may only be image format projects. The number of columns can be varied by change the the twitter bootstrap classes eg. "span3". Refer to the Twitter Bootstrap Responsive Gird for more info on these classes. The markup needed for this portfolio is shown below:
<div class="pe-container">

        <!--filter goes here-->


        <!-- Project Feed -->
        <div class="peIsotopeContainer">
            <div class="row-fluid">
                <div class="span12">


                    <!-- Columns Go Here -->


                </div>
            </div>
        </div>
    </div>
</div>
Defining Columns
The number of columns is determined by the number of columns used by each item. The max number of columns in any layout is 12. In the demo website, each item spans 3 columns, which mean there are 4 columns in the portfolio grid. A single row of grid Items would therefore look like this:
<!--single row-->
<div class="row-fluid">
    <!-- 4 x columns of 3span elements-->
    <div class="span3">portfolio item</div>
    <div class="span3">portfolio item</div>
    <div class="span3">portfolio item</div>
    <div class="span3">portfolio item</div>
</div>
Each row of the portfolio would need to be created like this wrapped in a div element with a class of row-fluid.
A 3 column layout with 2 rows would therefore look like this (again using the maximum of 12 columns):

2.2 Portfolio - Full Width Grid


This format uses the BigWig template's grid component, just like the gallery. Unlike the gallery however, here in the portfolio the grid is set to a regular layout. The grid cell size is set to 320x240px, however this can be other values if required. This size will vary slightly with different browser resolutions, to ensure a whole number of grid items fits in the browser window.

The markup required for this grid is as follows:
<div class="row-fluid">
    <div class="peIsotope portfolio">


        <!--filter goes here-->


        <div
            class="peIsotopeContainer peIsotopeGrid"
            data-cell-width="320"
            data-cell-height="240"
            data-sort="none"
            >


            <div class="row-fluid pe-gallery-thumbnails">
                <div class="span12">


                    <!--grid items go here-->


                </div>
            </div>
        </div>
    </div>
</div>
Parameter List
data-cell-widthCell width in pixels.
data-cell-heightCell height in pixels.

5.4 Flare - Gallery


Multiple images can be grouped in a gallery: when one of them is being displayed, it will be possible to navigate all of them without closing the lightbox. Navigation can happens in multiple ways: prev/next buttons, swipe gestures, mousewheel and keyboard. To group images in galleries, just add the data-flare-gallery attribute to the markup:
<a data-target="flare" data-flare-gallery="gallery1" href="01.jpg">Image 1</a>
<a data-target="flare" data-flare-gallery="gallery1" href="02.jpg">Image 2</a>
<a data-target="flare" data-flare-gallery="gallery1" href="03.jpg">Image 3</a>
All the above images will be grouped in "gallery1", you can have different galleries in the same page
Image navigation methods supported:

  1. Next / Previous UI Arrows
  2. Keyboard Left / Right Arrows
  3. Mouse Wheel Scroll
  4. Swipe Gestures (On Mobile & Tablet Devices)

If thumbs are showing, swipe gestures will control the thumbnail scrolling. Close the thumbnails to use swipe gestures on the main images.

5.7 Flare - Alternate Gallery, Shutter Effect


Flare also includes an alternate gallery mode using a black and white/zooming image transition: this mode is called shutter and can be activated using the data-flare-plugin attribute
<a
     data-target="flare"
     data-flare-plugin="shutter"
     data-flare-gallery="newGalleryName"
     data-flare-scale="fillmax"
     data-flare-thumb="img/content/small/01.jpg"
     data-flare-bw="img/content/big/01bw.jpg"
     href="img/content/big/01.jpg"
     >


    Fullscreen Shutter Gallery With Thumbs - Image 1

</a>
<a
     data-target="flare"
     data-flare-plugin="shutter"
     data-flare-gallery="newGalleryName"
     data-flare-scale="fillmax"
     data-flare-thumb="img/content/small/02.jpg"
     data-flare-bw="img/content/big/02bw.jpg"
     href="img/content/big/02.jpg"
     >


    Fullscreen Shutter Gallery With Thumbs - Image 2

</a>
Since shutter mode requires a black and white version of the image to do the transition, we need the extra parameter data-flare-bw to define it. The bw image is not mandatory and can be omitted, obviously in this case, the transition would skip the black and white effect part.
Black & White version of the main image and also the thumbnail image need to be prepared manually. this plugin does not automate this process.

12 BigWigtop

12.1 BigWig - The Fluid Grid


BigWig is a fully responsive template, this means that no matter what device a visitor is viewing your website on, the contents and the site layout will adjust to suit that devices default screen resolution. The default layout width of the BigWig template's fixed width pages are based on a maximum width of 940px. This will automatically remain centered on large resolution screens.The non-fixed width pages will simply expand to occupy the available space.

The fluid grid system uses percents for column widths instead of fixed pixels. The grid is a 12 column base CSS grid. A new row in the grid is denoted by the row-fluid class name while the width of a column or columns are denoted by a span# class name, where # represents a number of columns to span out of the base 12. So for example if you would like a div element to be half the width of its container you would use the following:
<div class="span6">...</div>
So following on from the above, the markup required for 2 column layout each column being half the size of its container, would be as follows:
<div class="row-fluid">
 <div class="span6">...</div>
 <div class="span6">...</div>
</div>
The total span number of columns inside one row-fluid element is 12. Also if the containing row-fluid element itself is wrapped by a div which only has a span of 6 columns, the max total span number of the columns is still 12. The reason for this is because the span number doesn't represent a fixed number of pixels but a percentage or ratio width. Therefore a wrapping div with span6 which contains a nested div with span6 would simply mean that the nested column would be 50% of its containing element, which in this case is a div of span6.
Here is an example of the markup required for nesting columns in the fluid grid:
<div class="row-fluid">
  <div class="span12">
   Level 1 of column

   <div class="row-fluid">
    <div class="span6">Level 2</div>
    <div class="span6">Level 2</div>
   </div>
  </div>
 </div>

12.2 BigWig - Responsive Layout


BigWig CSS works by detecting the resolution of the device which is viewing the website and then based on its size it serves up CSS styles specific to that resolution. The following table represents the supported devices their resolutions and the resulting column widths/layout.

SmartphonesResolution: 480px and below. Column Width: 280px
Smartphones to TabletsResolution: 481px to 767px. Column Width: 420px
Portrait TabletsResolution: 768px to 979px. Column Width: 42px
Desktop/LaptopResolution: 980px and above. Column Width: 60px
The CSS monitors the current device resolution using a technique called Media Queries. This is done inside the CSS stylesheet and looks like the following:
// Landscape phones and down
    @media (max-width: 480px) { ... }


    // Landscape phone to portrait tablet
    @media (max-width: 767px) { ... }


    // Portrait tablet to landscape and desktop
    @media (min-width: 768px) and (max-width: 979px) { ... }


    // Large desktop
    @media (min-width: 1200px) { ... }
All of the styles specific to a certain resolution are nested inside that specific media query (where the {...} occurs above. This means that if you wish to edit the CSS styles for a specific device layout, you will need to do so by locating the correct media query and editing only the CSS nested within it. Also note that the "large desktop" media query is not used in this template.

12.4 BigWig - Custom CSS, Finding


The main stylesheets for this template can be found in the css folder in the root package. The main stylesheet is named style.css. This template uses compressed CSS stylesheets (minified) in order to save on page loading times, and so if you wish to customize the CSS you will need to follow the guide below. Before you change an element's style you will first need to know what CSS rule that specific element is using to acquire its styles and then you can use that same rule to override those styles with your own. See the next section for instruction on how to find this information.

In the past we have had many support questions from buyers asking how to locate the CSS style statements which control the CSS for a specific element in the theme. So in response we have written this short section describing how to locate the correct CSS file and the exact line of code. Basically there are several options but here we will outline 2 of the easiest.

1- Use FireBug Firefox Add-on
This is a very handy little add on for the Firefox browser which can be found here https://addons.mozilla.org/en-US/firefox/addon/firebug/. Once installed, using the right mouse button whilst hovering over any item in the browser's main page area will bring up a context menu with the option Inspect Element shown at the bottom of the list. Clicking this option will open a window at the bottom of your browser. As standard this window will show the HTML structure on the left and the CSS styles on the right as they specifically pertain to the item over which you were hovering when you right clicked in the beginning. On the right side CSS panel the rules are in a scrollable list, the upper most rules being those greatest in hierarchy when it comes to applying the resulting style information to the item. On the right hand side of this panel inline with each individual CSS selector you will see a filename and a line reference number. With this information you can find the exact file and the exact line on which to change this element's CSS.

2- Google Chrome Browser
Like with Firebug the Google Chrome browser has similar element inspecting capabilities but in this case they are baked right into the browser itself and so no separate add-on is required. As described above the element selecting and inspecting process is the exact same, with the CSS stylesheet and exact line reference being easily accessible in the bottom right CSS styles panel. if you are a Chrome user, perhaps this method is more convenient for you.
Most modern browsers currently have a similar element inspector. Opera & Safari have a native version which can usually be accessed using a right click like Chrome. IE9 also has something similar but it need to be turned on first by clicking F12 key. To select element you then need to use the selector tool (arrow symbol).

12.5 BigWig - Custom CSS, Altering (Simple Version)


The following method can be used to modify the template's CSS styles. This method is only recommended when the CSS modifications are light and few in number, as overuse of internal CSS is contrary to best practices of CSS. The advantages of this method is that you do not need to touch the compressed stylesheets. To use this simple method, the following style tags need to be placed inside the head section of each HTML page in which you required the changes to appear. Then inside of these style tags, CSS selectors can be placed as normal.
<!-- modified CSS -->
<style>
    .taglineWrap{
        background:#eee;
        border-bottom:1px solid #ddd;
    }
</style>
Shown above is an example CSS selector to change the color of an element's background. To find out what kind of selectors are required to target a specific element in the template, refer to the previous section.
NOTE: Sometimes when using the above method for CSS modification, if a CSS style modification does not result in the expected style change even though the CSS selector syntax is correct, you may need to use the !important keyword in your CSS rules. This sill ensure the default rules are overwritten.

12.6 BigWig - Custom CSS, Altering (Advanced)


If you are intending to create a new skin for the template or if you wish to alter some of the existing CSS styles, a few things should be noted. For the purposed of page load speed the BigWig template uses compressed CSS stylesheets and Javascript. This means that instead of individually loading a separate stylesheet for different elements, the CSS stylesheets have been compressed into one file which is loaded by BigWig hence saving time with only one HTTP request. The consequences of this however is that if any changes are made to the uncompressed stylesheets, these will not be reflected in the compressed version without a re-compression being done. Therefore if you wish to change the css stylesheets you have 2 options.

  1. Use the "uncompressed" version of the template's files, found in the "alternate" folder in the package root. See the "Alternate" section above for more info.
  2. Create a new stylesheet and insert it after the minimised CSS stylesheet ensuring to use the same or stronger CSS selectors for each rule you wish to overwrite.
Inserting A New Stylesheet
<!-- compressed stylesheets -->
<link rel="stylesheet" href="theme/compressed/template.min.css" />


<!--insert new stylesheet here after the minified original-->
<link rel="stylesheet" href="css/new_style.css" />
How the Uncompressed Stylesheets Will look in the head section of the page
<!-- uncompressed stylesheets -->
<link href="css/bootstrap.css" rel="stylesheet" />
<link href="css/bootstrap-responsive.css" rel="stylesheet" />
<link href="css/style.css" rel="stylesheet" />
<link href="css/style-responsive.css" rel="stylesheet" />
The above statements are shown as examples only. The actual paths to the stylesheets should be checked before including in the head section. Check the "uncompressed" version of the template found in the "alternate/uncompressed" folder.

13 Mobiletop

BigWig has been optimized to work with Apple's mobile operating system iOS. The site has been tested on iOS6 primarily on the iPad2 and the iPhone 4. To ensure the template works well on iDevices, sometimes the CSS of the template needs to be tweaked or sometimes new rules need to be added to control specific behavior found only on the mobile version of Apple's Safari browser.


13.1 Mobile - Rollover Effects


Some of the interesting effects used in BigWig are seen or experienced by the user only when rolling over an element using their mouse. However when using a touch screen device such as the iDevices, the rollover state does not feature as prominently. It is still possible to see rollover effects as follows. In Apple's mobile Safari browser the way to activate a rollover state for an element is by Soft Clicking. This is simply a normal click, but as long as the element being clicked has a rollover state, then this state will be activated and a second click is then needed in order to visit the link or page or whatever is the default action of the link itself.

Sometimes however, rollover effects have been intentionally switched off in the CSS because they will not function correctly on a touch based device due to the different nature of a touch user interface from that of a mouse based one.



13.2 Mobile - Android


BigWig has been tested on the Android operating system using a Samsung Galaxy Tab 10.1. running the Ice Cream Sandwich version of the OS and also a Samsung Galaxy SII phone. While BigWig is working on these devices, the template requires quite a bit of cpu power to run smoothly and many mobile devices, especially older devices, may see reduced rendering speeds etc. There are some well known bugs in the Android OS relating to video playback. These are a flaw in the default browser of the OS and are not related to the template. Additionally it should be noted that despite having been optimized for Android ICS, this does not mean that this template will work flawlessly on all Android devices. The reasons for this are as follows:

  1. There are just a vast number of different devices from different manufacturers, out there all running android and all with varying hardware specs. Each manufacturer has also tweaked or skinned the OS in their own way. Consequently it is impossible to perform testing on all such devices and to allow for any slight differences there may be in the way each device renders content.
  2. Manufacturers of Android handsets and tablets are extremely slow in deploying updates to the OS. Consequently there are vast numbers of devices out in the wild which are running an array of different versions of the Android OS. Many of these devices may not have the hardware or software to run the template properly and so its impossible for us to promise full compatibility.
It is also worth noting that Android users may not enjoy as good an experience browsing this template as iOS users will. This is because at the time of writing the Android operating system does not allow for the experiencing of rollover states as iOS devices do.
